APM traces can become one of the largest cost drivers in your Datadog bill if not properly managed. A high-traffic application generating millions of traces per hour can quickly exceed your contracted span allocation, leading to significant overage charges. The key is to maintain observability while controlling costs through intelligent sampling.

In this section we'll explore how to control APM trace ingestion and indexing.

APM Quick Introduction
==========================================================

First, let's understand how APM tracing works in our lab environment.

Under [APM > Traces](https://app.datadoghq.com/apm/traces), you should see APM traces being generated by the application.

![Services Traces](../assets/services-traces.png)

For these traces to be reported to Datadog, the lab application was instrumented with auto-instrumentation, which requires no code changes.

On the **IDE** tab, open the **compose.yml** file to see an example of APM auto-instrumentation setup.

The **services.agent** config below enables the Datadog Agent to accept traces and trace metrics:
- `DD_APM_ENABLED=true` - Enable APM collection
- `DD_APM_NON_LOCAL_TRAFFIC=true` - Accept traces from containers

The **services.generator** config below enables the application to generate and send traces to the agent:
- The entrypoint uses `ddtrace-run python gen.py` to auto-instrument the Python application
- `DD_SERVICE=generator` - Service name for traces
- `DD_TRACE_SAMPLE_RATE=1.0` - Sample 100% of traces
- `DD_PROFILING_ENABLED=true` - Enable profiling data

**What Gets Traced Automatically:**
- Database operations (PostgreSQL queries)
- HTTP requests (External API calls)
- Cache operations (Redis interactions)
- Business logic processing
- Error traces

This lab application provides a safe environment to practice sampling techniques and observe their benefits. In production environments with much larger trace volumes, these sampling controls become essential for cost management.

APM Ingestion Control Methods 
==========================================================

The **Datadog UI** provides the primary interface for controlling APM trace ingestion. Although some configuration can be done directly on the Agent.

There are three types of methods available to adjust APM trace ingestion:

1. **Adaptive Sampling** (All services): Datadog automatically adjusts sampling to meet a target monthly volume (for example, based on your allotment or desired data volume in GB). This can be configured on the UI.

   ![Adaptive Sampling](https://learning-assets.datadoghq.com/apm-rate-limit-retention/lms/assets/service-ingestion-summary-web-store.png)

2. **Service-level or resource controls** (Per-service): Configure library-level sampling for specific services to reduce traces from noisy services. Supports both adaptive and custom sampling rates. Can be configured in the UI or directly in the tracing library. Sampling can target an entire service or specific resources within that service.

   ![Service Sampling](https://learning-assets.datadoghq.com/apm-rate-limit-retention/lms/assets/manage-ingestion-per-service.png)

3. **Agent-level Sampling Controls** (All services): Configure granular sampling at the Agent:
   - **Head-based sampling** (`ingestion_reason:auto`): Primary sampling at the root (head span) allowing to keep visibility of all the related spans.
   - **Error span sampling** (`ingestion_reason:error`): Ensures error traces are captured even when head-based sampling might drop some.
   - **Rare span sampling** (`ingestion_reason:rare`):  Using rule on caching unique tag combinations to send low-frequency traces in order to capture infrequent/anomalous execution paths.
   - **Custom rule sampling** (`ingestion_reason:rule`): Allows for custom rule sampling logic based on specific criteria.

   ![Agent Sampling](https://learning-assets.datadoghq.com/apm-rate-limit-retention/lms/assets/manage-ingestion-all-services.png)


To check the current configuration, go to [APM > Traces > Ingestion Control](https://app.datadoghq.com/apm/traces/ingestion-control).

All UI-based configurations leverage the Agent **Remote Configuration** feature (`DD_REMOTE_CONFIGURATION_ENABLED=true`) and are applied immediately without requiring an Agent restart.

While the UI is the recommended approach, other configuration methods are available for specific use cases (for example, OpenTelemetry). See the [Ingestion Control documentation](https://docs.datadoghq.com/tracing/trace_pipeline/ingestion_controls/).

Refer to the [sampling rules precedence](https://docs.datadoghq.com/tracing/trace_pipeline/ingestion_controls/?tab=remotely#sampling-rules-precedence) to understand the order of precedence when rules are configured in multiple locations.

![retention rules](https://learning-assets.datadoghq.com/apm-rate-limit-retention/lms/assets/trace-pipeline.png)

Reduce Trace Ingestion
==========================================================

Let's put this in practice.

Navigate to [APM > Traces](https://app.datadoghq.com/apm/traces) and adjust the view to visualize processed traces:
1. Set **Visualize as** to `timeseries`
2. Set **Group by** to `Service`
3. Set a proper timeline to see the change.

Notice a spike in traces across services: `generator`, `user-service`, `postgresql`, `redis`.

Apply a service-level ingestion control by following the steps below:
1. Navigate to [APM > Traces -> Ingestion Control](https://app.datadoghq.com/apm/traces/ingestion-control)
2. Under **Manage Ingestion Per-Service**, select the `generator` service
3. Click **Manage Ingestion Rate**
4. Select the `Custom sampling rate only` option
5. Select `All resources`
6. Set the sampling rate to `50%`
7. Select `Remotely` so the change is applied immediately (no application restart)
8. Validate your configuration matches the screenshot below
    ![Ingestion Control at service level](../assets/ingestion_control.png)
9. Click **Apply**

> [!NOTE]
> You could have also set the sampling rate for a specific resource of the `generator` service.

In couple minutes you can go back to [APM > Traces](https://app.datadoghq.com/apm/traces) to look at earliers timeseries graph to see the effect of your config changes.
![50% Traces](../assets/timeseries_50.png)

You an also hover **Traffic Breakdown** to see the percentage of traces that are not ingested:
![Discarded Traces](../assets/discarded_traces.png)

> [!NOTE]
> You can identify the docker container that is currently running and creating the spikes (`spiker`). By running `docker kill spiker` you can stop the spike. This demonstrates how quickly you can respond to cost spikes once detected. 

APM Retention Filters
========================================================

Retention filters decide which ingested spans are indexed (typically for 15 days), helping you balance cost and visibility.

Datadog provides an Intelligent Retention Filter that retains spans across environments, services, operations, and resources for different latency distributions, along with default filters that preserve visibility across services and endpoints and capture errors and high-latency traces. You can also define custom retention filters based on any span attribute or tag to retain the traces that matter most to your business.

See the [Datadog docs](https://docs.datadoghq.com/tracing/trace_pipeline/#retention-filters) for details.

> [!NOTE]
> If you want to do a more detail workshop on APM retention filter feel free to check this [learning course](https://learn.datadoghq.com/courses/apm-rate-limit-retention)

RUM Without Limits
=========================================================

RUM without Limits provides flexibility over your RUM session volumes by decoupling session data ingestion from indexing. This enables you to:

- Dynamically set retention filters from the Datadog UI without up-front sampling decisions or code changes
- Retain sessions with errors or performance issues while discarding less significant ones, such as sessions with few user interactions

Even if you retain only a fraction of your sessions, Datadog provides [performance metrics](https://docs.datadoghq.com/real_user_monitoring/rum_without_limits/metrics) for all ingested sessions, ensuring an accurate, long-term overview of application health and performance.

![RUM Without Limits](https://datadog-docs.imgix.net/images/real_user_monitoring/rum_without_limits/rum-without-limits-overview.png)

To configure RUM without Limits:

1. Navigate to [Digital Experience > Real User Monitoring > Applications](https://app.datadoghq.com/rum/list)
1. Select your application (in this case `Python web app`)
1. Configure SDK settings (found under `SDK Configuration`):
   - Set `sessionSampleRate` to 100%
   - Set `sessionReplaySampleRate` based on your observability needs
   - Ensure `traceContextInjection: sampled` is enabled
1. Configure retention filters in the Datadog UI:
   - Click `Retention Filters` under `PRODUCT SETTINGS`
   - Here you can adjust existing retention filters or add new ones based on your observability needs

![Retention Filters](../assets/RUM_without_limits.png)

Set up retention filter for RUM sessions
========================================

As you can observe under [Digital Experience > Real User Monitoring](https://app.datadoghq.com/rum/performance-monitoring) sessions are enabled and flowing into Datadog.

Try setting up retention filters to discard 75% of all sessions while keeping 100% of all error sessions. This demonstrates how RUM retention filters can drastically reduce ingestion costs.


> [!NOTE]
> You can also use [Single Step Instrumentation for RUM](https://docs.datadoghq.com/real_user_monitoring/application_monitoring/browser/setup/server/) on the server-side with supported web servers.
